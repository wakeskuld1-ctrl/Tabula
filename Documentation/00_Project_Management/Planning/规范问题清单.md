# 规范问题清单

## 目标
归档当前代码中不符合高内聚低耦合、开闭原则与可替换性等设计规范的问题，并为重构阶段提供可追踪依据。

## 后端（federated_query_engine）

### 1. 路由层与业务层耦合过高
- 表现：main.rs 中接口处理函数同时承担会话切换、SQL 拼接、上下文注册、元数据装载与响应格式化。
- 影响：修改任一逻辑容易触发连锁改动，测试难度高。

### 2. 单一函数职责过多
- 表现：execute_sql 同时负责日志、SQL 重写、自愈修复、执行与输出格式化。
- 影响：难以复用与隔离测试，违反单一职责。

### 3. 数据源扩展不满足开闭原则
- 表现：上传逻辑通过扩展名分支决定处理流程，新增数据源需改动主流程。
- 影响：扩展成本高，风险大。

### 4. 全局状态降低可替换性
- 表现：CacheManager 与请求合并注册表使用全局静态结构。
- 影响：测试替换困难，依赖隐式化。

### 5. 基础设施与业务逻辑混杂
- 表现：MetadataManager 直接依赖 DataFusion 与 SQLite 细节。
- 影响：难以替换存储或注册策略，抽象边界不清晰。

## 前端（frontend）

### 1. 组件职责过多
- 表现：GlideGrid 集成 UI、缓存、公式、网络、Undo/Redo、过滤、排序、粘贴处理等。
- 影响：组件难以维护与测试，职责高度耦合。

### 2. 业务规则散落
- 表现：列类型推断、列图标、后端交互与 UI 状态混在同一组件内。
- 影响：复用与替换成本高，违反高内聚低耦合。

## 关联代码位置
- main.rs：接口处理与查询逻辑聚合
- session_manager/mod.rs：会话与数据操作高度集中
- cache_manager.rs：全局状态与缓存策略混杂
- metadata_manager.rs：基础设施细节混入业务层
- GlideGrid.tsx：前端核心组件职责膨胀
