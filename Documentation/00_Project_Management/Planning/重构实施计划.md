# 重构实施计划

## 总体目标
在不改变外部行为的前提下，提高模块内聚、降低耦合，改进扩展性与可替换性，形成清晰的层次边界。

## 阶段一：后端接口层与服务层拆分

- 状态：完成
- 完成日期：2026-02-08

### 目标
- 将路由层与业务逻辑解耦
- 缩短单个 handler 函数长度

### 范围
- main.rs 中接口处理函数
- 引入服务层或 use-case 层承接业务逻辑

### 任务清单
- 任务一：建立最小分层骨架并迁移 health 接口
- 任务二：迁移 grid 查询链路
- 任务三：迁移 execute_sql 链路
- 任务四：迁移 upload 逻辑

### 任务一细化

#### 目标
- 建立 api 与 services 目录结构
- main.rs 仅负责路由绑定

#### 范围
- health 接口处理逻辑下沉

#### 验收点
- health 接口响应不变
- main.rs 不包含 health 业务逻辑

#### 风险
- 模块引用路径错误导致编译失败

### 验收点
- main.rs 仅保留路由与请求/响应编排
- 业务逻辑下沉到独立模块
- 单测通过，功能行为一致

### 跟进项
- 详见《重构跟进项.md》

## 阶段二：数据源与缓存体系可扩展化

### 目标
- 数据源接入满足开闭原则
- 缓存策略去全局状态化

### 范围
- upload 及数据源分支逻辑
- CacheManager 全局静态结构

### 任务清单
- 任务一：缓存体系解耦
- 任务二：数据源扩展机制
- 任务三：注册与元数据流程梳理
- 任务四：回归与性能验证

### 子任务细化

#### 任务一：缓存体系解耦
- 抽离 CacheManager 全局静态依赖
- 提供可注入的缓存接口与实现

#### 任务二：数据源扩展机制
- 统一数据源注册入口
- 规范数据源生命周期与配置

#### 任务三：注册与元数据流程梳理
- 统一注册流程与表名规范
- 梳理 metadata 写入与读取一致性

#### 任务四：回归与性能验证
- 回归用例覆盖关键链路
- 建立性能基线并对比

### 关键点
- 缓存依赖可注入、可替换
- 新增数据源不修改主流程
- 注册与元数据保持一致性

### 验收点
- 新增数据源不修改主流程
- 缓存依赖可注入、可替换
- 相关测试可独立执行
- 回归用例通过且性能不退化

## 阶段三：前端组件解耦与状态收敛

### 目标
- GlideGrid 拆分为可维护模块
- 类型推断与后端交互独立封装

### 范围
- 组件内网络、缓存、公式与 UI 状态
- 列类型推断与图标策略

### 验收点
- 核心渲染与业务逻辑分层
- 类型推断具备单独测试入口
- UI 行为保持一致

## 风险与注意事项
- 不改变现有 API 行为与返回结构
- 重构过程以小步提交、可回滚为原则
- 每阶段结束必须执行既有测试与关键回归用例
