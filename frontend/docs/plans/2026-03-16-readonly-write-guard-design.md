# 只读会话写入拦截与新建 Sheet 可靠性设计

> 日期：2026-03-16  
> 范围：Tabula 前端（默认会话只读、写入入口拦截、Sheet 创建可靠性）

## 背景

当前前端在默认只读会话下仍可能触发写入入口，导致用户输入后才被提示“无法保存”；同时首次创建 Sheet 偶发失败，导致从 Sheet2 起步。需要保证只读规则稳定生效，并修复新建 Sheet 的可靠性。

## 目标

1. **只读前置拦截**：默认会话不可写；用户触发“写入意图”时立即阻止并提示。  
2. **统一可写判定**：以 `sessionId` + `isReadOnly` 统一判断写权限。  
3. **Sheet 首次创建可靠**：避免“后台已建但前端认为失败”，保证 Sheet1 连续性。  
4. **不改变后端契约**：前端即可完成，后端改动最小化。

## 非目标

- 不改后端接口契约。  
- 不自动创建新 Sheet（仍需用户显式操作）。  
- 不引入复杂遮罩层（保持 alert 方式提示）。

## 方案概述（推荐方案 A）

**规则**：默认会话只读；当用户触发写入意图（保存/样式/编辑/粘贴/批量编辑/公式栏输入）时：
1. 立即 `alert`：“请先创建新 Sheet（session）再编辑/保存”
2. 阻止当前操作（不发写入请求）。

## 写入入口的覆盖范围

- 保存按钮（Toolbar）
- 样式按钮（加粗/斜体/下划线/对齐/颜色/背景）
- 公式栏输入（点击/键盘输入/提交）
- 表格编辑（键入/粘贴/填充/批量编辑）

## 数据流与判断

统一封装：

```
canWriteSession = Boolean(sessionId) && !isReadOnly
```

在每个“写入入口”触发前调用 `guardWrite()`：

- `false` → alert + return
- `true` → 继续原逻辑

## Sheet 首次创建失败的处理

问题可能来自两类情况：
1) 后端创建成功，但响应体缺少 `session_id`  
2) `from_session_id: null` 导致后端拒绝或产生异常响应

修复策略：
1. **最小证据采集**：记录 `create_session` 的 HTTP 状态与响应摘要（写入 debugInfo，不影响流程）。  
2. **缺 session_id 回退**：若响应缺 `session_id`，拉取 `/api/sessions` 并按 `session_name` 精确匹配新建项。  
3. **省略空字段**：当 `sessionId` 为空时，不发送 `from_session_id` 字段。

## 交互规范

- 只在“明确写入意图”发生时弹窗  
  - 例如：点击保存、点击样式按钮、开始输入公式  
  - 选中单元格时不弹窗
- 提示文案固定为：  
  **“请先创建新 Sheet（session）再编辑/保存”**

## 测试策略（TDD）

- **只读拦截**  
  - 当 `canWriteSession = false` 时：触发写入入口 → `alert`；不触发写入调用  
- **create_session 响应缺 `session_id`**  
  - 触发回退逻辑，能从 sessions 中找到新建项  
- **from_session_id 空值**  
  - 请求体不包含该字段

## 风险与缓解

- **提示过于频繁**：仅在写入意图触发时提示  
- **回退仍找不到新建项**：保持原错误提示，并建议重试  

## 结论

方案 A 兼顾规则一致性与实现成本，能在不依赖后端改动的前提下，快速修复当前体验问题与 Sheet1 跳号问题。
