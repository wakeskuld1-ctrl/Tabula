## **1. 核心背景与设计目标**

在 Tabula 中，数据源（DataSource）通常异构且分散（如 SQLite、CSV、Excel、Oracle）。每次查询都直接下推到源端拉取数据会面临以下痛点：
1.  **I/O 瓶颈**：CSV/Excel 解析开销巨大，重复解析浪费 CPU。
2.  **网络/连接瓶颈**：频繁建立数据库连接或跨网络传输会导致高延迟。
3.  **数据源压力**：高并发查询可能压垮业务数据库（如 Oracle/MySQL）。

因此，构建一套**“主动预热、分级缓存、智能淘汰”**的冷热数据管理系统是核心能力。本文档详细描述了已实现的“预热下推线”、“缓存清理线”及“防击穿机制”。

---

## **2. 预热与下推策略详解 (The Hot Line)**

这一条线的核心是**“走在用户前面”**。利用查询的时间局部性（Temporal Locality）和空间局部性（Spatial Locality），将被动等待变为主动出击。

### **2.1 智能预测性下推 (Predictive Pushdown)**

当优化器将 Filter（如时间范围 `2023-10-01 10:00:00`）下推到 `TableProvider` 时，系统执行以下逻辑：

#### **2.1.1 触发机制**
*   **识别模式**：在物理计划构建阶段，拦截 `TableScan` 的过滤条件。
    *   **时间连续型**：识别 `Time > T` 或 `Time BETWEEN T1 AND T2`。
*   **窗口预测算法**：
    *   如果用户查询区间长度为 `L`，系统默认预取 `L * 1.5` 的范围，或者向后预取固定 `Delta`（如 1 小时）。

#### **2.1.2 异步流水线设计**
为了不阻塞主查询，预热必须是完全异步的。已在 `sqlite.rs` 和 `oracle.rs` 中实现 `Sidecar` 模式：
*   **主线程**：立即返回用户请求的数据（L0 -> L1/L2）。
*   **Sidecar 线程**：后台异步将数据写入 Parquet 缓存（L1），并根据访问频率填充内存缓存（L2）。

### **2.2 维表全量预热 (Dimension Table Warmup)**
对于 CSV/Excel 等小文件，一旦检测到高频访问，直接触发全表加载并转码为 Arrow RecordBatch 驻留内存。

## **3. 缓存格式选型：为什么 Parquet 是核心？**

### **3.1 核心论断**
**L2（磁盘）缓存必须用 Parquet，L1（内存）缓存用 Arrow RecordBatch。**

| 特性 | CSV/Excel (原始格式) | Parquet (缓存格式) | 优势分析 |
| :--- | :--- | :--- | :--- |
| **类型系统** | 无 (全是字符串/Variant) | **强类型 (Schema)** | 避免每次读取都重新猜测类型 (Infer Schema)。 |
| **解析开销** | 极高 (文本解析/XML解压) | **极低 (二进制映射)** | 读取 Parquet 接近内存拷贝速度。 |
| **列式访问** | 不支持 (必须读整行) | **支持 (列式存储)** | IO 减少 90% 以上。 |

### **3.2 缓存转换架构 (Transcoding Cache)**
针对 CSV/Excel 数据源，已实现**“透明转码层”**：
1.  **Cache Miss**：读取原始文件 -> 解析 -> 写入 `shadow/{filename}.parquet` -> 返回 RecordBatch。
2.  **Cache Hit**：直接读取 `shadow/{filename}.parquet`。

## **4. 缓存清理与分层策略 (The Cold Line)**

为了防止 OOM，必须有严格的淘汰机制。我们已实现基于 **Log-Linear 评分** 和 **Eviction Pool** 的智能淘汰策略。

### **4.1 准入与生命周期 (Admission & Lifecycle)**

并非所有数据都能进入 L2 内存缓存，我们实施了严格的**“试用期” (Probation)** 机制：

*   **准入检查 (`should_promote_to_l2`)**：
    *   只有当数据块的访问次数 `access_count > 2` 或被标记为高优先级 (`priority > 1.0`) 时，才允许从 L1 (Disk) 提升到 L2 (Memory)。
    *   **目的**：防止“一次性扫描” (One-Hit Wonders) 污染宝贵的内存空间。

*   **分级 TTI (Time-To-Idle)**：
    *   **试用区 (Probation)**: 访问次数 < 2，空闲 **30秒** 后淘汰。
    *   **保护区 (Protected)**: 访问次数 ≥ 2，空闲 **5分钟** 后淘汰。

### **4.2 智能驱逐策略 (Eviction Strategy)**

当内存使用量超过阈值（默认系统内存的 70%）时，触发驱逐循环。我们采用了 **Log-Linear 评分 + 候选池** 算法，在性能与准确度之间取得平衡。

*   **评分公式 (Score)**：
    $$ Score = (\ln(Cost) - \ln(Size)) + 4.6 \times Priority + AgeFactor $$
    *   倾向于保留：计算代价大、体积小、高优先级、最近访问的数据。

*   **驱逐机制：Sampled Eviction with Candidate Pool**：
    *   **采样 (Sample)**: 每次循环随机抽取 **5** 个 Key。
    *   **候选池 (Pool)**: 维护一个容量为 **16** 的淘汰候选池。
    *   **流程**：
        1.  将新采样的 Key 加入池中。
        2.  池内按 Score 升序排序（分数越低越容易被淘汰）。
        3.  保留池中 Score 最低的 16 个元素。
        4.  淘汰池底最差的一个 Victim。
    *   **优势**：相比单纯的随机采样，引入池化机制能以全局视角“记住”最差的候选者，避免因运气不好误杀高价值数据。

## **5. 已实现的防击穿与规范化机制**

### **5.1 Singleflight (防缓存击穿)**
针对高并发场景（如 100 个请求同时查询同一个未缓存的 Key），实现了 **Singleflight / Request Coalescing** 模式：
*   **Leader**：第一个到达的请求成为 Leader，负责执行昂贵的 I/O 操作（查 DB 或文件转码）。
*   **Follower**：后续请求自动订阅 Leader 的结果通道（Broadcast Channel），挂起等待。
*   **覆盖场景**：
    *   **SQLite/Oracle 查询**：避免瞬间建立大量 DB 连接。
    *   **Excel/CSV 转码**：避免多个线程同时进行高 CPU 消耗的 Parquet 转换。

### **5.2 SQL 规范化 (Canonicalization)**
为了解决 `WHERE a=1 AND b=2` 与 `WHERE b=2 AND a=1` 生成不同 Cache Key 的问题，实现了 SQL 规范化逻辑：
*   **逻辑**：在生成 Key 前，对 SQL 谓词进行**字符串排序**和**括号平衡处理**。
*   **结果**：语义相同的 SQL 语句现在映射到同一个 Cache Key，极大提升了缓存命中率。

## **6. 验证结论 (Verification Results)**

我们对上述核心机制进行了单元测试验证，结果如下：

### **6.1 驱逐策略可靠性 (`test_eviction_reliability`)**
*   **测试场景**：设置极小的内存上限 (350 Bytes)，依次插入高价值 (High Cost)、低价值 (Low Cost) 和中价值数据。
*   **结果**：
    *   内存超限时，**低价值数据 (Victim) 被精准淘汰**。
    *   高价值数据即使体积相同，也因 Score 较高被保留。
*   **结论**：Eviction Pool 机制有效，能正确识别并保护核心数据。

### **6.2 规范化有效性 (`test_canonicalize_sql`)**
*   **测试场景**：输入乱序的 AND/OR 条件及嵌套括号 SQL。
*   **结果**：`a=1 AND b=2` 与 `b=2 AND a=1` 输出了完全一致的规范化字符串。

### **6.3 高负载与抗压验证 (`test_full_capabilities_stress`)**
*   **测试场景**：
    *   **高并发查询**：模拟 50 个并发线程同时请求同一数据，验证 SQL 规范化与 Singleflight 机制。
    *   **缓存洪水攻击 (Cache Flood)**：连续发起 200 个唯一查询（Unique Queries），每条查询结果约 50KB，试图撑爆 L2 内存缓存 (上限 5MB)。
*   **压测结果**：
    *   **L2 内存驱逐**：当内存使用接近上限 (5MB) 时，Eviction Pool 成功触发，日志显示 `[CacheManager] Evicted ... (Score: ...)`，内存占用回落至安全水位，未发生 OOM。
    *   **L1 磁盘驱逐**：当磁盘使用率超过阈值 (模拟环境 92%)，Log-Linear 策略成功识别并删除了旧的 Parquet 文件。
    *   **防击穿保护**：高并发阶段日志显示 `[SqliteExec] Cache Stampede Protection: Waiting for key ...`，确认 Singleflight 生效，大量请求被合并为单次 DB 查询。
*   **结论**：系统在持续高压下表现稳定，多级驱逐策略与并发控制机制均按预期工作。

## **7. 总结**

当前架构已从纯理论设计迈向了工程落地：
1.  **分层清晰**：L0 (Source) -> L1 (Parquet/Disk) -> L2 (Arrow/Memory)。
2.  **保护完善**：通过 Singleflight 防止击穿，通过 Probation 防止污染，通过 Pool 防止误杀。
3.  **性能验证**：关键路径均已覆盖单元测试，确保了策略的确定性和可靠性。
