# 代码架构原则审查记录（SOLID / DRY / KISS / YAGNI / 迪米特 / 组合优于继承）

> **[2026-03-09] 记录原因：补齐一次面向可维护性的架构体检结论；记录目的：为后续重构提供明确优先级与落地路径**

## 1. 审查范围

- 后端：`federated_query_engine/src`
- 前端：`frontend/src`
- 工作区入口与项目结构：根目录 `Cargo.toml`、`README.md`

## 2. 总体现状

- 后端存在“内联实现 + 分层实现”并行状态：`main.rs` 与 `api/* + services/*` 功能重叠。
- 前端存在“当前实现 + 预研实现”并行状态：`App.tsx` 为入口，`App.univer.tsx` 未接入。
- 核心会话能力集中在超大文件 `session_manager/mod.rs`，职责边界偏重。

## 3. 关键问题清单（按原则归类）

### 3.1 SOLID

#### S（单一职责）问题

- `main.rs` 同时承担启动、路由、上传、注册、执行、计划、日志等职责，变更面过大。
- `session_manager/mod.rs` 同时承担会话生命周期、持久化、更新、刷盘与测试，模块内聚不足。

#### O（开闭原则）问题

- 数据源分发依赖多处 `if/else` 判断 `source_type`，新增数据源需要在多个位置同时改动。

#### D（依赖倒置）问题

- 业务服务直接依赖具体 `AppState` 及其深层字段，缺少稳定抽象层，替换实现成本高。

### 3.2 DRY（避免重复）

- SQL 执行与计划逻辑在 `main.rs` 与 `services/execute_service.rs` 存在重复实现。
- 前端列名转换逻辑在 `GlideGrid.tsx` 与 `utils/formulaRange.js` 各维护一份。

### 3.3 KISS（保持简单）

- `App.tsx` 保留了历史 FortuneSheet 状态与注释代码，当前实际渲染路径是 `WasmGrid`，认知成本偏高。

### 3.4 YAGNI（不做暂时不需要）

- `App.univer.tsx` 当前未被入口使用，保留在主路径会增加维护噪音与依赖复杂度。

### 3.5 迪米特法则（最少知道原则）

- 服务层出现较多链式访问（例如通过 `state` 连续下钻），增加耦合与联动修改风险。

### 3.6 组合优于继承

- 项目未出现明显“继承滥用”问题。
- 主要问题是“组合粒度不足”：`SessionManager` 仍接近 God Object，应拆为可组合子模块。

## 4. 整改优先级与建议

### P0（优先立即执行）

1. 统一后端入口：路由只保留 `api/* -> services/*` 一条执行路径。
2. 收敛重复逻辑：删除 `main.rs` 中与 `execute_service` 等重复的业务实现。

### P1（短周期执行）

1. 抽象数据源工厂/注册表：替代多处分支判断，新增数据源只需注册一次。
2. 在 `AppState` 上提供窄接口：减少服务层对深层对象结构的直接依赖。

### P2（中周期执行）

1. 拆分 `SessionManager`：按生命周期、持久化、更新、自动刷盘分模块组合。
2. 前端统一工具函数：提取列名转换与类型推断公共能力，消除重复实现。
3. 清理未启用路径：对 `App.univer.tsx` 明确“启用计划”或“迁移出主路径”。

## 5. 风险与收益评估

- 风险：重构过程中可能出现路由行为偏差、SQL 自愈链路回归、会话状态一致性问题。
- 收益：降低变更耦合、减少重复缺陷、提升可测试性与新功能接入效率。

## 6. 建议的验证用例

1. 路由回归：上传、注册、执行、计划、健康检查接口全量回归。
2. 会话回归：切换会话、自动分叉、批量更新、自动刷盘与重启恢复。
3. SQL 回归：字段自愈重写、大小写重写、分页过滤排序组合查询。
4. 前端回归：表格加载、公式列显示、筛选排序、编辑与撤销重做流程。

## 7. 结论

- 当前代码已经具备较完整能力，但在“职责边界”和“重复实现”上存在明显结构性债务。
- 建议按 **P0 -> P1 -> P2** 逐步收敛，先统一执行路径，再抽象扩展点，最后做模块化拆分。

## 8. 落地执行计划（按周推进）

> **[2026-03-09] 计划原因：将评审结论转换为可执行任务；计划目的：控制重构风险并保证每周可验收**

### Week 1：P0 路径统一与重复收敛

#### 目标

- 后端对外行为不变，内部仅保留 `api/* -> services/*` 一条业务路径。
- 去除 `main.rs` 与 `services` 的重复执行/计划逻辑。

#### 任务清单

1. 路由对齐：将 `main.rs` 路由全部改为调用 `api` 层 handler。
2. 逻辑收敛：删除 `main.rs` 中重复的执行、计划、上传、注册业务函数。
3. 启动瘦身：`main.rs` 仅保留启动、装配、路由注册、静态资源托管。
4. 回归验证：执行上传、注册、执行、计划、健康检查全链路回归。

#### 验收标准

- 代码中不存在第二套 SQL 执行/计划实现。
- API 回归通过，关键接口返回结构与重构前一致。

### Week 2：P1 抽象扩展点与依赖降耦

#### 目标

- 新增数据源只需注册一次，不再改多处分支。
- 服务层不直接下钻 `AppState` 深层结构。

#### 任务清单

1. 新建数据源注册表（Factory/Registry）并接管 `source_type` 分发。
2. 在 `AppState` 提供窄接口（如 `cache_manager()`、`table_store()`）。
3. 将 `register_service`、`upload_service`、`grid_service` 改为依赖窄接口。
4. 补充单元测试：覆盖新增数据源注册、未知类型处理、接口返回一致性。

#### 验收标准

- 新增一个数据源时仅需新增注册项，不改业务分发主流程。
- 服务层链式访问显著减少，可通过代码审查清单检查通过。

### Week 3：P2 模块拆分与前端收敛

#### 目标

- `SessionManager` 按职责拆分为组合模块。
- 前端移除未启用路径噪音，统一公共工具。

#### 任务清单

1. `SessionManager` 拆分为：会话生命周期、持久化、更新、自动刷盘 4 个子模块。
2. 前端提取列名转换与类型推断公共函数，替换重复实现。
3. 明确 `App.univer.tsx` 去留策略：启用排期或迁移出主路径。
4. 回归验证：会话切换、自动分叉、批量更新、编辑撤销重做、公式列展示。

#### 验收标准

- `session_manager/mod.rs` 文件体量显著下降，职责边界清晰。
- 前端重复工具函数收敛为单一实现，业务行为不回退。

## 9. 执行节奏与质量门禁

### 提交节奏

- 每完成一个可验收小任务即提交一次，避免超大变更难以回滚。
- 每周末统一做一次端到端回归并输出简短周报。

### 质量门禁

- Rust：`cargo fmt --all -- --check`、`cargo clippy --all-targets --all-features -- -D warnings`、`cargo test`。
- Frontend：`npm run build`，并执行关键交互冒烟脚本（加载、筛选、编辑、公式列）。
- 文档：架构文档与接口行为变更保持同步，避免“代码更新、文档滞后”。
