基于 Rust 与 Apache Arrow 的分布式查询引擎架构深度分析报告

1. 执行摘要与系统基础

在现代数据密集型应用中，传统的查询执行模型正面临前所未有的挑战。随着数据规模的指数级增长和实时分析需求的迫切性提升，基于 Java 或 C++ 的传统架构在内存安全、并发模型以及序列化开销方面显现出局限性。本报告旨在提供一份详尽的架构分析，深入探讨基于 Rust 编程语言和 Apache Arrow 内存模型的分布式查询引擎（以 Apache DataFusion 和 Ballista 为蓝本）的内部构造。

报告将系统地解构该架构的三个核心维度：系统架构（System Architecture）、类与特征设计（Class & Trait Design）以及数据流机制（Data Flow Mechanics）。分析的核心论点在于，Rust 的所有权模型与 Arrow 的列式内存布局相结合，不仅解决了“内存安全”与“高性能”之间的传统权衡，还通过“零拷贝（Zero-Copy）”原则重塑了分布式计算的数据传输效率。

通过对逻辑规划、物理执行、优化器规则以及分布式调度器的深度剖析，本报告揭示了现代查询引擎如何通过模块化设计实现高度的可扩展性。报告还将详细阐述 TableProvider 和 ExecutionPlan 等核心抽象如何支持异构数据源的无缝集成，并展示数据如何在没有任何序列化开销的情况下，从存储层流向计算层，最终通过网络传输。

2. 系统宏观架构图解与分层解析

现代 Rust 查询引擎的架构并非单体巨石，而是采用了一种高度模块化、分层解耦的设计。这种设计遵循编译器架构的经典范式（前端、中端、后端），并针对数据处理进行了特定的适配。

2.1 架构分层详解

2.1.1 接口层与会话管理 (API & Session Layer)

整个系统的入口是 SessionContext。在 Rust 的实现中，这一层不仅是用户交互的界面（提供 SQL 和 DataFrame API），更是系统状态的持有者。SessionContext 封装了 SessionState，其中包含了注册的 Catalog（目录）、TableProvider（表提供者）、ScalarUDF（标量用户自定义函数）以及配置选项 SessionConfig。这一层的设计体现了“配置与执行分离”的原则，允许用户在不重启引擎的情况下动态调整执行参数（如并发度、内存限制等）。

2.1.2 逻辑规划层 (Logical Planning Layer)

当 SQL 查询字符串进入系统后，首先由 sqlparser-rs 进行词法分析和语法分析，生成抽象语法树（AST）。随后，SqlToRel 转换器将 AST 转换为未解析的逻辑计划（Unresolved Logical Plan）。在此阶段，系统会利用 Catalog 中的元数据来解析表名和列名，生成解析后的逻辑计划。逻辑计划是一棵由 LogicalPlan 枚举节点构成的树，每个节点代表一种关系代数操作（如 Projection, Filter, Aggregate, Join）。此层完全独立于物理执行细节，仅描述“要做什么”而非“怎么做”。

2.1.3 优化器层 (Optimizer Layer)

优化器是查询引擎的“大脑”。在 DataFusion 中，优化器由一系列 OptimizerRule 组成。这些规则按顺序应用于逻辑计划，执行包括谓词下推（Predicate Pushdown）、投影下推（Projection Pushdown）、常量折叠（Constant Folding）和公共子表达式消除等操作。这一层的设计极具扩展性，允许开发者注入自定义的优化规则（例如，针对特定数据源的几何优化或 AI 模型推理优化）。

2.1.4 物理规划层 (Physical Planning Layer)

经过优化的逻辑计划被传递给物理规划器（Physical Planner）。该组件负责将逻辑算子转换为具体的物理执行算子（ExecutionPlan）。这一转换过程涉及具体的算法选择，例如将逻辑 Join 转换为 HashJoinExec 或 SortMergeJoinExec。物理规划器还必须处理数据分布（Partitioning）和排序（Ordering）要求，插入必要的 CoalesceBatchesExec（合并批次）或 RepartitionExec（重分区）算子以满足后续算子的输入契约。

2.1.5 分布式调度层 (Distributed Scheduling - Ballista)

在分布式模式下，架构扩展为 Scheduler-Executor 模型。

  • Scheduler（调度器）： 负责接收逻辑计划，并将其根据“Shuffle 边界（Shuffle Boundaries）”切割成多个执行阶段（Stages）。每个阶段包含一组可以并行执行的任务（Tasks）。调度器维护集群状态，通过 gRPC 将任务分发给活跃的 Executor。
  • Executor（执行器）： 负责实际的计算。它接收序列化的物理计划（Protobuf 格式），在本地线程池中执行，并将结果存储在内存或磁盘上。
  • Shuffle 管理： 不同执行阶段之间的数据交换通过 Arrow Flight 协议进行。这是一种基于 HTTP/2 的高性能数据传输标准，允许 Executor 之间直接流式传输 Arrow RecordBatch，避免了传统的 JDBC/ODBC 序列化开销。

2.3 架构洞察与分析

该架构最显著的特征是零拷贝原则的贯彻。从存储层读取 Parquet 文件时，数据被直接解码为 Arrow 格式；在内存中处理时，所有算子（Filter, Project, Aggregate）都直接操作 Arrow 数组；在分布式 Shuffle 过程中，Arrow Flight 协议直接传输这些数组的内存布局。这种端到端的一致性消除了传统大数据系统中常见的“序列化-反序列化（SerDe）”瓶颈，这是 Rust 查询引擎性能优越的根本原因之一。

此外，可扩展性设计体现在系统的每一个环节。开发者可以通过实现 TableProvider 特征来接入新的数据源（如 Iceberg 或 Delta Lake），通过实现 OptimizerRule 来添加新的优化逻辑，甚至通过实现 PhysicalPlanner 来完全重写物理计划的生成过程。这种设计使得 DataFusion 不仅仅是一个查询引擎，更是一个构建数据库系统的“工具包”。

3. 类图设计与面向对象分析

在 Rust 中，虽然没有传统面向对象编程中的“类（Class）”概念，但通过结构体（Struct）和特征（Trait）的组合，实现了一套强大的多态系统。理解这些核心特征的继承与实现关系，是掌握引擎扩展机制的关键。

3.1 核心特征详解

3.1.1 TableProvider 特征

TableProvider 是数据源抽象的核心。它屏蔽了底层存储的差异，向上层提供统一的接口。无论是本地 CSV 文件、S3 上的 Parquet 数据集，还是远程的 MySQL 数据库，对于查询引擎来说，都是一个实现了 TableProvider 的对象。

  • schema(&self) -> SchemaRef: 返回表的 Schema。这是逻辑规划阶段进行类型检查的基础。
  • scan(...) -> Result>: 这是连接逻辑层与物理层的桥梁。它接收投影（Projection）、过滤器（Filters）和限制（Limit），并返回一个能够执行读取操作的物理执行计划。
  • supports_filters_pushdown(...): 这是一个关键的优化钩子。它允许数据源声明自己能处理哪些过滤条件。例如，PostgreSQL 数据源可以声明它能完全处理 SQL WHERE 子句，从而避免引擎在内存中再次过滤，极大减少数据传输量。

3.1.2 ExecutionPlan 特征

ExecutionPlan 代表物理执行图中的一个节点。它对应于数据库内部的物理算子（Operator）。

  • execute(...) -> Result: 核心执行方法。它接受分区索引，并返回一个异步数据流（Stream）。这种基于 Future 和 Stream 的设计使得执行引擎能够天然地与 Rust 的 tokio 异步运行时集成，实现高并发的非阻塞 I/O。
  • output_partitioning(): 描述输出数据的分区方式（如哈希分区、轮询分区）。这对于分布式调度至关重要，决定了是否需要插入 Shuffle 操作。
  • output_ordering(): 描述输出数据的排序状态。优化器利用此信息来消除冗余的排序操作（例如，在需要有序输入的归并连接前，如果数据源已排序，则无需再次排序）。

3.1.3 LogicalPlan 枚举

与 ExecutionPlan 使用特征不同，LogicalPlan 通常实现为枚举（Enum）。这种设计利用了 Rust 强大的模式匹配（Pattern Matching）功能，使得优化器编写规则时非常方便。

  • Variants: Projection, Filter, Aggregate, Sort, Join, TableScan, Limit 等。每个变体都包含其所需的子计划和表达式（Expr）。

3.3 类设计深度解析

Rust 的 trait 系统在这里发挥了至关重要的作用。与 C++ 的虚函数继承相比，Rust 的 trait object（dyn ExecutionPlan）提供了类似的动态分发能力，但结合泛型（Generics）和关联类型（Associated Types），可以在编译期进行更激进的优化。

例如，ListingTable 是一个非常通用的实现，它组合了 ObjectStore（处理 S3/本地文件 I/O）和 FileFormat（处理 Parquet/CSV 编解码）。这种组合优于继承的设计（Composition over Inheritance）使得添加对 Azure Blob Storage 的支持只需实现一个新的 ObjectStore，而无需修改 ListingTable 或 ParquetExec 的代码。这种高度解耦的设计模式是 Rust 生态系统库（Crate）互操作性强的核心原因。

4. 数据流图与执行管道分析

理解数据如何在系统中流动，对于性能调优和系统理解至关重要。数据流分析不仅涉及逻辑上的转换，还涉及物理上的内存布局变化和网络传输。

4.1 数据流阶段详解

4.1.1 阶段一：从 SQL 到 逻辑计划 (Input -> Logical)

用户提交 SQL 查询。sqlparser 将其解析为 AST。此时数据尚未涉及，仅涉及元数据。SqlToRel 绑定器会检查 Catalog，确保表 t 存在且包含列 a。如果列名错误，流程在此终止。此阶段产生的逻辑计划包含了完整的语义信息，但未包含物理执行细节。

4.1.2 阶段二：逻辑优化与谓词下推 (Optimization)

假设查询是 SELECT name FROM users WHERE age > 25 AND region = 'US'。优化器遍历逻辑计划树。

  1. PushDownFilter: 优化器检查 TableProvider 是否支持过滤。对于 Parquet 文件，age > 25 可以利用 Parquet 的统计信息（Min/Max）直接跳过不包含符合条件数据的 RowGroup。TableProvider 返回 Exact，表示该过滤器被下推到底层，DataFusion 不需要再在内存中执行此过滤（或者只需要对剩余数据块执行）。
  2. ProjectionPushDown: 只有 name 列被请求。优化器通知扫描算子只读取 name 列的数据块，极大地减少了 I/O 带宽消耗。

4.1.3 阶段三：物理执行与向量化计算 (Physical Execution)

在执行阶段，ExecutionPlan 树被构建。根节点调用 execute，触发级联调用直至叶子节点（如 ParquetExec）。

  1. 读取: ParquetExec 异步从磁盘读取压缩的 Page，解码为 Arrow 的 RecordBatch。注意，这里读取的是列式数据，每一列是一个连续的内存块。
  2. 过滤: 如果有未下推的过滤器，FilterExec 会接收 RecordBatch。它不会逐行检查，而是利用 SIMD 指令计算一个布尔掩码（Boolean Mask），然后利用 Arrow 的 filter 内核（Kernel）生成一个新的、经过切片的 RecordBatch。这个过程是零拷贝的（Zero-Copy Slicing），只是更新了引用计数和偏移量。
  3. 聚合: HashAggregateExec 接收批次数据，更新内部的哈希表。为了性能，哈希表的键和值也是以向量化格式存储的。

4.1.4 阶段四：分布式 Shuffle (Shuffle & Transmission)

在分布式场景下，数据需要重分区。

  1. Shuffle Write: 上游 Executor 计算每行数据的哈希值，将其分发到不同的内存缓冲区。当缓冲区满时，序列化为 Arrow IPC 格式并写入磁盘或发送到网络。
  2. Shuffle Read: 下游 Executor 通过 Arrow Flight 请求数据。Flight 服务端直接将内存中的 Arrow 批次映射到网络 socket，客户端直接从 socket 读取并映射回内存。整个过程绕过了类似 Protobuf 或 JSON 的高昂序列化/反序列化开销。

4.3 数据流机制分析

该数据流设计的核心优势在于批处理（Batching）与向量化（Vectorization）。与传统的“一次处理一行（Tuple-at-a-time / Volcano Model）”模型相比，处理 RecordBatch（通常包含 4000-8000 行）极大地分摊了函数调用的开销，并提高了 CPU 指令缓存（I-Cache）的命中率。此外，Rust 的 async 生态（Tokio）使得 I/O 操作（读取 Parquet）可以与 CPU 计算（过滤、聚合）并行流水线化，最大化硬件利用率。

特别值得注意的是，数据流中的内存管理采用了基于 MemoryPool 的配额机制。每个算子在申请内存前必须向 MemoryPool 预订。如果内存不足，系统会触发溢出机制（Spilling），将部分中间数据写入磁盘（由 DiskManager 管理），这保证了查询引擎在处理超过内存大小的数据集时不会崩溃（OOM）。

5. 优化器深度解析与 AI 集成

除了基础的规则优化，现代 Rust 查询引擎还在探索更高级的优化技术。

5.1 统计信息与成本模型

ExecutionPlan 特征包含一个 statistics() 方法，返回关于数据集的估计（行数、字节大小、列的 Min/Max 值）。物理规划器利用这些统计信息来做出成本决策。例如，在决定 Join 策略时，如果一侧表非常小（且统计信息准确），规划器会选择 BroadcastHashJoin（将小表广播到所有节点），而不是昂贵的 HashJoin（需要 Shuffle 重分区）。

5.2 动态过滤器 (Dynamic Filter Pruning)

在执行 Join 操作（如 Fact 表 Join Dimension 表）时，引擎可以在运行时构建一个 Bloom Filter（布隆过滤器），并将其从 Build 侧（维度表）“反向”传递给 Probe 侧（事实表）的扫描算子。这使得事实表的扫描算子能够利用这个运行时生成的过滤器过早地丢弃数据，即使在逻辑计划阶段无法预知过滤条件。这种技术在 DataFusion 中被称为动态过滤，能带来数倍的性能提升。

5.3 AI4DB：学习型优化器

随着 AI 的发展，Rust 查询引擎也开始集成机器学习技术。正如文献所述，研究者正在探索在 DataFusion 中实现“学习型基数估计（Learned Cardinality Estimation）”和“学习型索引（Learned Indexes）”。通过实现自定义的 OptimizerRule，可以将传统的直方图基数估计替换为调用轻量级神经网络模型的推理过程。Rust 的 FFI 能力使得它可以轻松调用 PyTorch 或 TensorFlow 的模型（通过 tch-rs 等库），或者直接使用 Rust 原生的机器学习库（如 linfa 或 SmartCore）。

6. 结论与未来展望

本报告对基于 Rust 和 Arrow 的分布式查询引擎进行了全面的架构解构。通过分析可以看出，该架构的成功并非偶然，而是基于几个关键的设计选择：

  1. 语言选择的战略优势： Rust 的内存安全特性消除了这一类系统中常见的段错误和并发竞争条件，同时提供了与 C++ 匹敌的性能。
  2. 数据标准的统一： 采纳 Apache Arrow 作为核心内存格式，使得计算层、网络层（Flight）和存储层（Parquet）达成了空前的一致性，使得“零拷贝”成为现实。
  3. 极致的模块化： 通过 TableProvider、ExecutionPlan 和 OptimizerRule 等特征的松耦合设计，系统具备了极强的演进能力。

未来展望：

  • 流批一体化： RecordBatchStream 的抽象天然支持流式数据。我们预计将看到更多基于此架构的流处理引擎（如 RisingWave, Arroyo）出现，它们复用大部分批处理的物理算子。
  • 硬件加速的普及： 鉴于 Arrow 内存布局对 GPU 友好，未来结合 rapids-rs 等库，将特定物理算子（如 Join, Aggregation）卸载到 GPU 将成为常态，且无需昂贵的数据格式转换。
  • 存储与计算的进一步融合： 随着 ObjectStore 接口的标准化，查询引擎将更深入地与云原生存储（S3, Azure Blob）集成，利用本地 NVMe 缓存（如 Foyer 库）来掩盖网络延迟，实现无状态计算节点的极致弹性。

综上所述，DataFusion 和 Ballista 不仅代表了查询引擎技术的现状，更指引了未来高性能数据系统的构建方向：安全、标准、模块化且极致高效。

组件层级 Rust/DataFusion 实现 传统实现 (Java/Spark) 优势分析
内存模型 Apache Arrow (Off-heap, Columnar) Java Objects / Tungsten (Row/Column Hybrid) 消除 GC 暂停，SIMD 友好，内存利用率高
数据传输 Arrow Flight (Zero-copy) Java Serialization / Kryo 极低的 CPU 序列化开销，高吞吐量
执行模型 Vectorized Execution (Async/Poll) Volcano / Codegen 批处理分摊开销，Async 更好地利用 IO 等待时间
扩展性 Trait-based (TableProvider, OptimizerRule) Interface / Abstract Classes Rust Trait 提供零成本抽象，无运行时虚函数开销 (在泛型化时)
并发模型 Tokio (Work-stealing async runtime) Thread Pool / Executor Service 极轻量级任务切换，适应高并发连接
