# AI 架构方案（Rust + 云 API + 本地文件）

> **[2026-03-15] 变更原因：按需求整理 AI 架构方案文档；变更目的：明确 Excel 前端直连大模型的数据分析链路与组件选型。**

## 1. 背景与目标
- 背景：当前系统提供 Web 版 Excel 体验（React + Vite），数据来源以本地磁盘上的 Parquet/Lance 为主。
- 目标：在不返回 SQL/代码给用户的前提下，让 Excel 前端一键触发 AI 分析并获得可解释、可复现的结果。
- 约束：不使用 SQL 字符串；云 API 为主要模型调用方式；本地文件存储为当前主场景。

**备注（2026-03-15）：**
- 采用“受控 DSL/AST → 执行计划”的方式替代 SQL，可降低幻觉与注入风险。
- 面向普通用户不返回任何代码/SQL，结果仅以图表/结论/摘要呈现。

## 2. 总体架构与分层
### 2.1 组件分层
1) **Excel Web 前端**
- AI 面板：自然语言输入、范围/列选择、结果回写。  

2) **Rust API 层**
- 统一入口：`/api/ai/analyze`
- 负责：鉴权、限流、审计、缓存、任务编排。

3) **LLM 编排层（Rig）**
- 负责：意图识别、工具调用编排、生成 DSL/AST。  

4) **语义检索层（LanceDB）**
- 存储：表/字段语义、历史分析摘要、业务术语向量索引。

5) **结构化执行层（DataFusion）**
- 接收 AST → 构建 LogicalPlan/DataFrame → 执行 Parquet/Lance 读取与聚合。

6) **数据层（本地文件）**
- Parquet/Lance 文件；未来可迁移到对象存储。

**备注（2026-03-15）：**
- 选型核心：Rig 编排 + LanceDB 语义检索 + DataFusion 执行引擎。
- 后续迁移对象存储只影响“数据层适配器”，不改变上层协议。

### 2.2 数据流（高层）
```mermaid
flowchart LR
  "Excel Web 前端" --> "Rust API /api/ai/analyze"
  "Rust API /api/ai/analyze" --> "Rig 编排"
  "Rig 编排" --> "语义检索（LanceDB）"
  "Rig 编排" --> "AST 校验器"
  "AST 校验器" --> "DataFusion 计划执行"
  "DataFusion 计划执行" --> "Parquet/Lance 本地文件"
  "DataFusion 计划执行" --> "结果汇总"
  "结果汇总" --> "Rig 解释生成"

```

**备注（2026-03-15）：**
- 严格把控“AST 校验 → 计划执行”关口，杜绝直接执行 LLM 输出。
- 解释生成与数据执行解耦，便于审计与复现。

## 3. DSL/AST 设计（V1）
### 3.1 设计原则
- 只允许白名单操作（scan/filter/group/agg/sort/limit）。
- 禁止任意 UDF 与外部网络访问。
- AST 必须可序列化、可审计、可回放。

**备注（2026-03-15）：**
- AST 作为唯一执行入口，避免 SQL 字符串与 Rust 代码生成。

### 3.2 AST 示例（JSON）
```json
{
  "dataset": "table_sales_2026",
  "select": ["region", "amount"],
  "filter": [{ "col": "region", "op": "=", "value": "East" }],
  "group_by": ["region"],
  "aggs": [{ "func": "sum", "col": "amount", "as": "total_amount" }],
  "sort": [{ "col": "total_amount", "dir": "desc" }],
  "limit": 50
}
```

**备注（2026-03-15）：**
- `dataset` 必须来自注册表；`select/filter/agg` 字段必须存在且类型兼容。
- `limit`、`row_budget` 由后端强制注入默认值，避免大表爆炸。

### 3.3 校验规则
- 字段存在性 + 类型兼容（数值列才能 sum/avg）。
- 操作白名单（不支持 JOIN/窗口函数时直接拒绝）。
- 资源预算：最大行数/最大执行时长/最大内存。

**备注（2026-03-15）：**
- 校验失败必须返回“可读错误 + 修正建议”。

## 4. 语义索引与元数据链路
### 4.1 元数据来源
- Excel 上传：抽取表名、列名、类型、样本值、统计信息。
- 未来 MySQL：同步 schema + 指标口径 + 字段描述。

**备注（2026-03-15）：**
- 目前无数据字典，需在上传时自动生成“轻量语义元数据”。

### 4.2 LanceDB 索引结构（建议）
- 表级文档：表说明、来源、更新时间、行数、主键。
- 字段级文档：字段名、类型、样本、可能含义、同义词。
- 分析摘要：历史分析结论 + 关键字段引用。

**备注（2026-03-15）：**
- 语义检索只做“提示与纠偏”，不直接决定执行逻辑。

## 5. LLM 工具与编排
### 5.1 工具集合（建议）
- `get_schema(dataset)`：返回字段与类型
- `search_semantics(query)`：基于 LanceDB 的语义检索
- `build_ast(user_intent, schema, hints)`：生成 DSL/AST
- `execute_ast(ast)`：调用 DataFusion
- `summarize(result)`：生成可解释自然语言

**备注（2026-03-15）：**
- `execute_ast` 前必须经过后端校验器，Rig 只负责编排。

### 5.2 结果返回结构
- `summary`: 文字解释
- `table_preview`: 前 N 行或聚合结果
- `confidence`: 置信度/风险标记
- `audit_id`: 审计标识

**备注（2026-03-15）：**
- 对用户隐藏 DSL/AST，仅在后台审计可见。

## 6. 安全与治理
- 只允许访问注册的数据集与路径。
- 资源限制：执行超时、最大行数、并发限流。
- 审计记录：AST、数据集版本、执行时长、调用者。

**备注（2026-03-15）：**
- 统一审计口径可为后续监管/排障提供依据。

## 7. 错误处理
- 语义错误：字段不存在/类型不匹配 → 明确提示可替代字段。
- 执行错误：超时/内存不足 → 建议降采样或分批。
- 模型错误：输出非法 AST → 回滚并要求重试。

**备注（2026-03-15）：**
- 错误提示不暴露内部路径或敏感信息。

## 8. 测试策略（建议）
- AST 校验单测：字段、类型、白名单约束。
- 集成测试：Excel → API → AST → 执行 → 回写。
- 性能测试：大文件抽样、聚合性能基线。

**备注（2026-03-15）：**
- 优先保障“可复现性”与“失败可解释性”。

## 9. 里程碑（建议）
- M1：AST 设计 + 校验器 + DataFusion 执行
- M2：LanceDB 语义检索接入 + 语义纠偏
- M3：前端 AI 面板联调 + 审计/限流完善

**备注（2026-03-15）：**
- 里程碑与当前项目节奏对齐，可按需拆分。
