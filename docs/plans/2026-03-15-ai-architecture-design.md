# AI 架构设计记录（方案 A）

> **[2026-03-15] 变更原因：用户确认方案A；变更目的：归档本次 AI 架构方案决策与边界约束。**

## 一、需求摘要
- 目标：Excel 前端直接调用大模型分析本地 Parquet/Lance 数据。
- 约束：不使用 SQL 字符串；云 API 为主；目前数据在本地磁盘。

**备注（2026-03-15）：**
- 不向普通用户返回 SQL/代码。

## 二、方案对比
### 方案 A（推荐）
- LLM → 受控 DSL/AST → DataFusion DataFrame/LogicalPlan
- 优点：安全可控、可复现、审计友好
- 缺点：需要维护 AST → 计划的映射层

### 方案 B
- LLM → SQL → 解析/白名单 → DataFusion
- 优点：表达力强
- 缺点：SQL 转义复杂，安全边界更难控制

### 方案 C
- LLM → Polars LazyFrame
- 优点：开发快
- 缺点：与 DataFusion/LanceDB 统一治理成本高

**备注（2026-03-15）：**
- 最终选择方案 A，符合“禁 SQL”的约束。

## 三、关键决策
1) 使用 Rig 进行 LLM 编排
2) 使用 LanceDB 维护语义索引
3) 使用 DataFusion 执行 AST

**备注（2026-03-15）：**
- 结构上保留后续对象存储迁移能力。

## 四、范围与非目标
- 范围：Excel Web 前端 → Rust API → DSL/AST → DataFusion
- 非目标：复杂跨源 JOIN、任意 UDF、暴露 SQL/代码给用户

**备注（2026-03-15）：**
- V1 仅覆盖 scan/filter/group/agg/sort/limit。

## 五、风险与缓解
- 风险：AST 与真实业务语义偏差
- 缓解：语义索引纠偏 + 字段类型/白名单校验

**备注（2026-03-15）：**
