# 公式列设计方案（方案1）

日期：2026-02-16  
作者：AI 协作  
范围：前端公式列交互、后端公式列计算与元数据持久化

## 背景
当前表格支持公式计算，但缺少“插入公式列”的一体化体验与持久化能力。需求目标是：用户可在列级别定义公式，列中每行展示计算结果，且该列禁止逐单元格编辑。

## 目标
- 支持“插入公式列”入口与公式输入弹层。
- 公式列只读，禁止逐单元格编辑。
- 公式列元数据持久化，支持刷新与随数据变化自动更新。
- 兼容“首行为表头/首行为非表头”的列名映射。

## 非目标
- 不实现跨表复杂依赖图的自动重算调度。
- 不在后端落地物化公式结果。

## 方案选型
### 方案1（采用）
后端保存公式列元数据（raw + sql），查询时将公式列作为计算列拼接到 SELECT；前端负责公式输入与列只读提示。

**优点**
- 公式结果与筛选/排序一致。
- 无需物化，成本稳定。

**缺点**
- 需要维护表达式到 SQL 的映射与校验。

### 方案2（不采用）
前端本地公式引擎计算，不参与后端查询。

**缺点**
- 只覆盖当前页，过滤/排序不一致。

### 方案3（不采用）
公式结果物化写入列，依赖更新时批量重算。

**缺点**
- 更新成本高，难保证一致性。

## 数据流
1. 前端触发“插入公式列”，提示输入公式表达式（如 `C*F`）。
2. 前端将表达式解析为 SQL 片段并提交后端。
3. 后端保存公式 marker 到 `column_default_formulas_json`。
4. 拉取网格数据时，SQL SELECT 对应列替换为计算表达式。
5. 返回前端时标注公式列元信息，前端设置只读与公式栏展示。

## 接口与数据结构
### 新增接口
- `POST /api/update-column-formula`
  - 请求：`{ table_name, session_id?, col_idx, formula_raw, formula_sql }`
  - 响应：`{ status, message }`

### 网格数据响应补充
- `formula_columns`: `[{ index, name, raw_expression, sql_expression }]`

## 前端交互
- 右键列头：新增“插入公式列”入口。
- 插入流程：弹窗输入公式，提示“插入公式列不允许编辑单个单元格”。
- 只读：公式列所有单元格设置只读。
- 公式栏：选中公式列单元格时显示原始公式字符串。

## 后端处理
- 公式 marker 采用 JSON 存储：
  - `{ "kind": "formula", "raw": "...", "sql": "..." }`
- 网格查询时根据 marker 生成 SELECT 列清单。
- 更新单元格时阻止编辑公式列。

## 测试策略
- 单元测试：SQL 拼接包含公式列表达式。
- 接口测试：更新公式接口校验空表达式报错。
- 前端回归：插入公式列后只读、公式栏展示、刷新行为。

## 风险与应对
- SQL 表达式非法：前端与后端同时校验，返回友好错误。
- 列名映射错误：通过“首行为表头/非表头”模式映射逻辑覆盖。
